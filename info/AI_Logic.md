### 1. Динамический расчет WIP-лимитов (Dynamic WIP Limits)

Эта функция опирается на закон Литтла (Little's Law) из теории массового обслуживания, адаптированный для гибких методологий. Суть в том, чтобы система сама подсказывала оптимальное количество задач в работе, при котором команда не будет простаивать, но и не захлебнется в многозадачности.

**Необходимые данные (из вашей модели):**
* Средний Throughput (пропускная способность) за последние 4–8 недель.
* Медианный Cycle Time (перцентиль P50) за тот же период.
* Текущее значение WIP.
* Массив `transitions` для вычисления времени нахождения в каждом статусе.

**Бизнес-логика и формулы:**
1.  **Расчет глобального WIP-лимита системы:** Применяется базовый закон Литтла: $WIP = Throughput \times Cycle Time$. Поскольку Throughput у вас считается в неделях, а Cycle Time — в днях, необходимо привести их к единой размерности.
2.  **Формула глобального лимита:** `Target System WIP = (Weekly Throughput / 7) * (Cycle Time P50)`. Полученное значение округляется до целого числа. Это идеальное количество активных задач для всей системы.
3.  **Распределение лимита по статусам (колонкам):** ИИ вычисляет долю времени, которую задачи исторически проводят в каждом конкретном статусе. Например, если средний `Cycle Time` равен 10 дням, и из них задачи лежат в `ctStart` (Разработка) 6 дней, а в "Ревью" 4 дня, то пропорция составляет 60% на 40%.
4.  **Формула локального лимита:** `Target Column WIP = Target System WIP * (Average Time in Status / Total Cycle Time)`. Округление до целого.

**Паттерны поведения ИИ-агента:**
* Если текущий суммарный WIP превышает `Target System WIP` более чем на 20%, ИИ выдает предупреждение о риске снижения Throughput и роста Cycle Time.
* Агент еженедельно пересчитывает лимиты и предлагает их скорректировать (например: «За последний месяц ваша пропускная способность выросла на 15%. Предлагаю увеличить WIP-лимит колонки "Ревью" с 3 до 4»).

### 2. Детектор узких мест (Bottleneck Analysis)

Эта функция фокусируется на микро-уровне потока. Ее задача — находить статусы, где работа накапливается быстрее, чем обрабатывается, или где задачи "стареют" сверх нормативных показателей.

**Необходимые данные (из вашей модели):**
* Записи переходов `transitions` (в частности, поля `status` и `enteredAt`).
* Конфигурация воркфлоу `WorkflowConfig` (порядок статусов).
* Данные метрики Aging WIP (возраст задач в работе).

**Бизнес-логика и формулы:**
1.  **Расчет времени в статусе (Time in Status - TiS):** Для каждой исторически завершенной задачи рассчитывается время пребывания в каждом статусе воркфлоу.
2.  **Формула TiS:** `TiS = Время входа в следующий статус - enteredAt текущего статуса` (в миллисекундах, затем конвертируется в дни).
3.  **Формирование исторического бейзлайна:** ИИ вычисляет перцентили P50 и P85 для TiS каждого статуса за последние 3 месяца. Это формирует "нормальное" поведение системы.
4.  **Детекция застоев (Queue Build-up):** ИИ проверяет количество активных задач в каждом статусе. Если в статусе $N$ задач больше, чем в статусе $N+1$ (следующем по воркфлоу) на заданный коэффициент (например, в 2.5 раза), триггерится алерт "Накопление очереди".
5.  **Детекция аномального старения (Aging Anomaly):** ИИ сканирует текущие незавершенные задачи. Если время нахождения задачи в *текущем конкретном статусе* превышает исторический показатель `TiS P85` для этого статуса, срабатывает алерт (в отличие от Aging WIP, который смотрит на общий возраст от `ctStart`).
