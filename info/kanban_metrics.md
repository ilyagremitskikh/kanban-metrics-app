# Kanban Metrics App

* **Описание:** Документ содержит описание бизнес-логики и методик расчёта метрик приложения.
* **Версия документа:** 1.0.
* **Дата:** 06.04.2026.

---

## 1. Общее описание приложения
Kanban Metrics App — это веб-приложение, предназначенное для анализа потока работ команды с использованием данных из Jira. Приложение способно рассчитывать такие ключевые канбан-метрики, как Lead Time, Cycle Time, Throughput и WIP, а также строить графики, осуществлять вероятностное прогнозирование с помощью метода Монте-Карло и помогать в приоритизации задач по фреймворку RICE. 
* **Технологический стек:** Проект написан с использованием React 19, TypeScript, Vite, Chart.js и Tailwind CSS.

### 1.1. Архитектура потока данных
Данные в приложении проходят через несколько этапов:
* **Настройки (Settings):** Пользователь вводит URL вебхука n8n, выбирает нужный диапазон дат, проект Jira и конкретные типы задач (также поддерживается произвольный JQL-запрос).
* **Загрузка данных (API):** Система отправляет POST-запрос на указанный вебхук n8n. Чтобы обойти лимиты API, диапазон дат разделяется на месячные отрезки. В ответ приходит массив задач (Issue), включающий историю изменения их статусов. После получения задачи дедуплицируются по их уникальному ключу.
* **Определение воркфлоу (Workflow Detection):** Приложение автоматически объединяет типы задач в группы, если у них схожие наборы статусов (пересечение более 50%). Для каждой такой группы вычисляются границы метрик: `ltStart`, `ltEnd`, `ctStart` и `ctEnd`. При необходимости пользователь может скорректировать эти границы вручную.
* **Расчёт метрик (Metrics):** Базируясь на загруженных задачах и границах воркфлоу, система рассчитывает Lead Time, Cycle Time, Throughput и WIP — как индивидуально для каждой задачи, так и в виде агрегированных данных.
* **Визуализация (UI):** Итоговые метрики выводятся на экран в виде информационных карточек (отображаются средние значения и перцентили P50/P85/P95), scatter-графиков, столбчатых диаграмм пропускной способности, а также в формате таблицы Aging WIP и результатов прогноза Монте-Карло.

## 2. Модель данных

### 2.1. Задача (Issue)
Основной единицей данных выступает задача из Jira, которая содержит следующие атрибуты:
* **Ключ (`key`):** Уникальный идентификатор задачи в Jira (например, CREDITS-9011).
* **Название (`summary`):** Текстовое описание задачи.
* **Тип (`type`):** Категория задачи (например, User Story, Задача, Ошибка или Техдолг).
* **Дата создания (`created`):** Временная метка в формате ISO, указывающая на момент создания.
* **Текущий статус (`currentStatus`):** Актуальный статус, в котором находится задача.
* **История переходов (`transitions`):** Массив данных о перемещениях задачи по статусам с указанием времени.
* **Категория статуса (`statusCategory`):** Внутренняя категория Jira (new, indeterminate, done).
* **Резолюция (`resolution`):** Итог работы (Done, Won't Fix, Duplicate и т.д.).

### 2.2. Переход (Transition)
Каждая запись о переходе фиксирует момент, когда задача получила определённый статус:
* **Статус (`status`):** Имя статуса (например, "Разработка" или "Готово").
* **Время входа (`enteredAt`):** Точный момент получения этого статуса (в формате ISO timestamp).

### 2.3. Конфигурация воркфлоу (WorkflowConfig)
Воркфлоу задаёт рамки для расчёта метрик для определённых групп задач. Конфигурация включает:
* **Типы задач (`issueTypes`):** Список типов, к которым применяется данный воркфлоу (например, User Story).
* **Статусы (`statuses`):** Жизненный цикл в виде упорядоченного списка (например: Идея → Бэклог → Разработка → Готово).
* **Границы Lead Time:** `ltStart` обозначает вход задачи в поток (например, "Идея"), а `ltEnd` — её завершение (например, "Готово").
* **Границы Cycle Time:** `ctStart` фиксирует начало активной работы (например, "Разработка"), а `ctEnd` — завершение активной разработки (например, "Готова к Prod").

Система автоопределения различает два основных типа процессов:
1.  **User Story:** Включает этапы открытия/исследования (например, "Идея", "Проработка идеи"). Границы устанавливаются так: `ltStart` = Идея, `ctStart` = Разработка, `ltEnd` = Готово.
2.  **Задачи/Ошибки:** В этих процессах отсутствуют discovery-статусы. Границы: `ltStart` = Бэклог, `ctStart` = Разработка, `ltEnd` = Готово, `ctEnd` = Готова к Prod.

## 3. Метрики: Lead Time

### 3.1. Определение
Lead Time (LT) отражает полное время жизни задачи, начиная с момента её попадания в поток и до самого завершения. Эта метрика измеряется в днях и включает как время активной работы, так и время простоя в очередях.

### 3.2. Формула
Расчёт производится по формуле: `LT = (Время входа в ltEnd − Время входа в ltStart) / 86 400 000 мс`.
* Используется timestamp первого перехода задачи в её конечный статус (например, "Готово") и первого перехода в начальный статус ("Идея" или "Бэклог").
* В случае, если задача миновала статус `ltStart`, точкой отсчёта становится дата её создания.

### 3.3. Особенности расчёта
* Если задача ещё не дошла до статуса `ltEnd`, метрика не рассчитывается (Lead Time = null).
* Если данные некорректны (время завершения меньше или равно времени начала), метрика также равна null.
* Значение выводится в виде дробного числа (например, 5.73 дня).
* **Пример:** Если задача создана и перешла в статус "Идея" 1 марта, а статус "Готово" получила 8 марта, её Lead Time составит ровно 7.0 дней.

## 4. Метрики: Cycle Time

### 4.1. Определение
Cycle Time (CT) измеряет исключительно время активной работы над задачей (от начала разработки до её завершения), исключая время ожидания перед взятием в работу. Метрика также рассчитывается в днях.

### 4.2. Формула
Формула расчёта: `CT = (Время входа в resolvedEndStatus − Время последнего входа в ctStart) / 86 400 000 мс`.
* Конечный статус (`resolvedEndStatus`) определяется гибко: если задача была в статусе `ctEnd`, берётся он; если этот этап был пропущен (задача пошла по сокращённому пути), в качестве запасного варианта (fallback) используется `ltEnd`.
* Для времени начала берётся самый поздний timestamp перехода в `ctStart`, произошедший до завершения задачи. Это позволяет правильно считать время, если задача возвращалась в работу после ревью.

### 4.3. Особенности расчёта
* Fallback-механизм предотвращает выдачу пустого значения (null) для задач, перепрыгнувших через этап (например, сразу из "Разработки" в "Готово").
* Использование последнего входа в статус разработки гарантирует, что при возвратах задачи отсчёт времени обновится.
* Метрика будет равна null, если задача так и не побывала в статусах `ctEnd` или `ltEnd`, либо не входила в `ctStart` до своего завершения.
* **Пример:** Задача перешла в "Разработку" 3 марта, ушла на "Ревью" 5 марта, затем вернулась в "Разработку" 6 марта и перешла в "Готова к Prod" 9 марта. Cycle Time будет считаться от последнего возврата (6 марта) и составит 3.0 дня.

## 5. Метрики: Throughput (Пропускная способность)

### 5.1. Определение
Throughput отображает количество задач, которые команда успешно завершает за одну неделю, что служит индикатором её реальной производительности.

### 5.2. Алгоритм расчёта
* Датой завершения считается самый ранний переход задачи в статус `ltEnd`.
* Из расчёта исключаются задачи, которые были закрыты без фактической доставки результата (например, с резолюциями "Won't Fix", "Duplicate", "Невозможно воспроизвести" или "Не является ошибкой").
* Для единообразия дата завершения приводится к понедельнику текущей недели.
* Задачи группируются понедельно для подсчёта их количества.
* Важным этапом является заполнение пустых недель (между первой и последней) нулевыми значениями — без этого среднее значение не учитывало бы простои команды.

### 5.3. Агрегированные показатели
Для этой метрики рассчитываются: среднее значение за неделю (mean), максимальное количество завершённых задач в неделю (max) и общее число завершённых задач за анализируемый период.

## 6. Метрики: WIP (Work in Progress)

### 6.1. Определение
WIP показывает количество задач, над которыми ведётся активная работа в данный момент. Задача попадает в WIP, только если соблюдаются три условия:
1.  Она ещё не завершена (текущий статус не равен `ltEnd`).
2.  Она не переведена в терминальный нерабочий статус (например, "Отменена", "Архив", "Установлено").
3.  Она не находится в статусе предварительной работы (pre-work).

### 6.2. Определение pre-work статусов
Статус классифицируется как pre-work, если он находится перед `ctStart` в настроенном списке воркфлоу.
* Статус `ltStart` (например, "Идея") всегда считается предварительной работой.
* Если оба сравниваемых статуса присутствуют в списке, система просто сопоставляет их индексы.
* Если текущий статус в списке не найден, система применяет консервативный подход и также относит его к pre-work.

### 6.3. Терминальные нерабочие статусы
Статусы, указывающие на прекращение работы без фактической реализации, исключаются из расчёта WIP. К ним относятся: "Отменена", "Архив" и "Установлено" (для специфических процессов).

### 6.4. WIP по статусам
Приложение может показывать распределение WIP по конкретным статусам (например, 5 задач в Ревью и 3 в Разработке), что помогает визуализировать узкие места, где скапливается работа.

## 7. Расчёт перцентилей

### 7.1. Метод
Приложение рассчитывает перцентили P50, P85 и P95, используя стандартный для статистики метод линейной интерполяции (R-7).

### 7.2. Формулы
* Сначала вычисляется индекс: `idx = (P / 100) × (N − 1)`, где `P` — это нужный перцентиль, а `N` — общее количество значений.
* Если индекс получается целым числом, берётся точное значение из массива.
* Если индекс дробный, применяется линейная интерполяция между двумя соседними значениями: `result = arr[floor(idx)] + (arr[ceil(idx)] − arr[floor(idx)]) × (idx − floor(idx))`.

### 7.3. Интерпретация
* **P50 (медиана):** Показывает, что половина задач завершается быстрее этого времени (используется для типичных ожиданий).
* **P85:** 85% задач завершаются быстрее (используется для обязательств об уровне сервиса — SLE).
* **P95:** 95% задач укладываются в это время. Значения выше считаются аномальными выбросами.

## 8. Aging WIP (Возраст задач в работе)

### 8.1. Определение
Эта метрика анализирует количество дней, которое каждая активная задача уже провела в работе. Она помогает выявить скрытые блокировки и задачи, рискующие нарушить допустимые сроки.

### 8.2. Расчёт возраста
Возраст рассчитывается по формуле: `(Текущее время − Время последнего входа в ctStart) / 86 400 000 мс`. Расчёт применяется только к задачам, входящим в текущий WIP, и использует тот же принцип последнего входа в статус `ctStart`, что и Cycle Time.

### 8.3. Цветовая индикация
Для наглядности задачи окрашиваются в зависимости от их возраста по отношению к перцентилям Cycle Time:
* **Зелёный (Возраст ≤ CT P50):** Сроки выполнения задачи в пределах нормы.
* **Жёлтый (CT P50 < Возраст ≤ CT P85):** Задача подходит к границе обязательств (SLE).
* **Красный (Возраст > CT P85):** Задача превышает нормальные сроки и требует немедленного внимания.

В интерфейсе отображается до 40 самых "старых" задач, отсортированных по убыванию возраста.

## 9. Прогнозирование методом Монте-Карло

### 9.1. Общий принцип
Приложение использует метод Монте-Карло для создания вероятностных прогнозов, опираясь на исторические данные о пропускной способности (Throughput). Система выполняет 10 000 симуляций, случайным образом выбирая исторические значения недельного throughput (включая пустые недели).

### 9.2. Режим "Количество задач" (Items Mode)
* Отвечает на вопрос: *"Сколько недель потребуется, чтобы завершить N задач?"*.
* Алгоритм задаёт целевое число задач. В каждой симуляции случайный throughput прибавляется к счётчику, пока он не достигнет или не превысит цель. Система фиксирует количество затраченных недель (с ограничением в 520 недель или 10 лет) и выводит результаты по перцентилям P50, P85, P95.

### 9.3. Режим "Дата" (Date Mode)
* Отвечает на вопрос: *"Сколько задач мы успеем завершить к дате X?"*.
* Алгоритм высчитывает количество полных недель до указанной даты. В каждой симуляции суммируется случайный throughput для каждой из этих недель. Результат также выдаётся в виде перцентилей P50/P85/P95.

### 9.4. Режим "Очередь" (Queue Mode)
* Отвечает на вопрос: *"Когда будет завершена каждая задача из приоритизированного бэклога?"*.
* Пользователь указывает размер текущего WIP и список задач в очереди. Система высчитывает общее количество позиций (WIP + очередь) и в каждой симуляции суммирует случайный throughput по неделям. Задачи из WIP обрабатываются в первую очередь, затем — задачи из бэклога. Для каждой задачи формируется индивидуальный прогноз (P50, P85, P95) дат её завершения.

### 9.5. Формирование результата
Итоги 10 000 симуляций интерпретируются через перцентили:
* **P50:** Результат будет не хуже этого значения с вероятностью 50%.
* **P85:** Результат будет не хуже с вероятностью 85%.
* **P95:** Результат будет не хуже с вероятностью 95%.

## 10. RICE-скоринг

### 10.1. Определение
RICE — это фреймворк для оценки и приоритизации задач, который использует четыре параметра для вычисления итогового балла.

### 10.2. Формула
Расчёт: `RICE Score = (Reach × Impact × Confidence / 100) / Effort`. Полученное значение округляется до целого числа.

### 10.3. Компоненты
* **Reach (Охват):** Любое положительное число, означающее количество пользователей, которых затронет изменение в месяц.
* **Impact (Воздействие):** Степень влияния на пользователя (допустимые значения: 0.25, 0.5, 1, 2, 3).
* **Confidence (Уверенность):** Уверенность команды в оценках (25%, 50%, 80%, 100%).
* **Effort (Трудозатраты):** Оценка объёма работы в story points (от 0.5 до 40 с шагом 0.5).

### 10.4. Интерпретация Impact
Влияние (Impact) оценивается по шкале: 3 — критическое (Massive), 2 — высокое (High), 1 — среднее (Medium), 0.5 — низкое (Low), 0.25 — минимальное (Minimal).

### 10.5. Визуальная индикация
Задачи ранжируются по убыванию итогового балла и подсвечиваются разными цветами: зелёным (верхние 66% — высокий приоритет), жёлтым (средние 33–66%) и серым (нижние 33% — низкий приоритет).

### 10.6. Интеграция с Монте-Карло
Задачи, оцененные по RICE, можно напрямую экспортировать в режим "Очередь" метода Монте-Карло для прогнозирования дат их завершения с учётом выставленных приоритетов.

## 11. Визуализация

* **Scatter-графики (Lead/Cycle Time):** Это точечные диаграммы, где каждая завершённая задача представлена отдельной точкой. Ось X показывает дату завершения, а ось Y — значение метрики в днях. На график нанесены горизонтальные линии перцентилей (P50, P85, P95).
* **Столбчатая диаграмма Throughput:** Отображает столбцы по неделям (начиная с понедельника). Высота столбца соответствует количеству завершённых задач, а горизонтальная линия показывает среднее значение.
* **Карточки метрик:** Четыре информационные панели с общими показателями (Lead Time, Cycle Time, WIP, Throughput). Внутри выводится среднее значение, перцентили и объём проанализированных данных.
* **Виджет Predictability Score (Процент прогнозируемости):** Отображает метрику стабильности процесса, основанную на коэффициенте вариации (CV) значения Cycle Time за скользящее 12-недельное окно. Метрика показывает, насколько предсказуемо завершаются задачи: >85% (Высокая), 50-85% (Средняя), <50% (Низкая). Снабжён мини-графиком (sparkline) для отслеживания тренда стабильности.

### 11.1. Визуальная классификация (UI/UX)
Приложение использует эргономичную систему бейджей и индикаторов на уровне всего интерфейса (в таблицах "Задачи" и "Aging WIP"):
* **Иконки типов задач:** Вместо текста применяются строгие SVG-иконки: документ (User Story), жук (Ошибка/Bug), шестерёнка (Техдолг), задача со стикером (Задача), позволяющие быстро считывать тип тикета.
* **Цветовое кодирование статусов:** Статусы воркфлоу автоматически красятся в соответствующие их этапу оттенки (опираясь на пайплайны Discovery & Delivery): Фиолетовый (Идея/Аналитика), Синий (Бэклог/Ready), Индиго (Разработка), Оранжевый (Review/Тестирование), Зеленый (Готово/Prod), Красный/Серый (Отмена/Архив).
* **Сохранение состояния вкладок:** Приложение использует механизм "горячего" переключения (фоновое CSS-скрытие слоев, в частности вкладки RICE). Это гарантирует, что несохраненные оценки и конфигурации списков задач не слетят, если пользователь переключится для анализа метрик.

## 12. Хранение данных
Для сохранения данных между сессиями приложение задействует `localStorage` браузера:
* `km_settings`: хранит настройки проекта (URL вебхука, даты, проект, типы задач, JQL).
* `km_workflows`: сохраняет конфигурации воркфлоу (как автоопределённые, так и ручные настройки).

Сами задачи (issues) не сохраняются локально. Они загружаются в оперативную память при каждом новом обращении к API.

## 13. Внешние интеграции

### 13.1. n8n Webhook (основные данные)
Интеграция работает через POST-запросы к вебхуку n8n и поддерживает два режима:
* **Standard:** передаются параметры проекта, типов задач, дополнительных условий и дат. Для обхода лимитов Jira API этот диапазон дат запрашивается по частям (чанками) длиной в месяц.
* **Custom JQL:** передаётся кастомный JQL-запрос и диапазон дат.

### 13.2. RICE API
Интеграция для оценок работает через два эндпоинта: `GET /webhook/rice-scoring` (для загрузки списка задач с текущими баллами) и `POST /webhook/rice-score-update` (для сохранения новых оценок).
