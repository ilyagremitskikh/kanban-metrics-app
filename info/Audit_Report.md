# Аудит бизнес-процессов и реализации расчета метрик Kanban Metrics App

**Дата аудита:** 06.04.2026.
**Краткий вывод:** Выявлены **критические логические ошибки** между задуманными бизнес-процессами, архитектурой загрузки данных в n8n и реализацией расчетов метрик на фронтенде. Данные ошибки приводят к ложным статистическим результатам (Throughput, Cycle Time, WIP). Приложение в текущем виде нельзя использовать для бизнес-отчетности.

---

## 1. Фатальные архитектурные ошибки выборки данных (n8n API)

### 1.1. Ошибка фильтрации дат (JQL: `created >=`)
* **Где найдено:** `n8n Workflow - Kanban Metrics - Jira Data.json` (Узел `Prepare JQL`).
* **Суть несостыковки:** Ограничение окна анализа базируется на фильтрах `created >= dateFrom` и `created <= dateTo`.
* **Последствия:** При попытке проанализировать метрики (например, за март 2026) система выберет только те задачи, которые были **заведены** в марте. Долгие "тяжелые" задачи или User Story, висевшие в бэклоге с декабря и закрытые в марте, **не попадут в расчет**. Это искусственно занижает Throughput, катастрофически искажает расчет WIP (старые задачи просто пропадают с доски) и ломает прогноз Монте-Карло.
* **Рекомендация:** Для метрик Kanban критически важно собирать задачи по дате *завершения* или всех *активных* вне зависимости от даты создания. JQL нужно переписать: `statusCategory != Done OR (resolved >= "dateFrom" AND resolved <= "dateTo")`.

### 1.2. Потеря поля Resolution (Резолюция) и ложный Throughput
* **Где найдено:** В n8n webhook (узлы `Fetch Jira Issues` и `Transform Data`), фронтенд файл `src/lib/metrics.ts`.
* **Суть несостыковки:** Документация (п. 5.2) требует: *«Из расчёта исключаются задачи, которые были закрыты без фактической доставки... (Won't Fix, Duplicate)»*. Фронтенд это проверяет (`issue.resolution`). **НО** n8n вообще не скачивает это поле из Jira: указано строго `fields: "summary,issuetype,created,status"`.
* **Последствия:** Свойство `resolution` всегда `undefined`. Задачи, отклоненные командой как дубликаты, засчитываются как успешная поставка! Это критически завышает производительность, обманывая бизнес.
* **Рекомендация:** Добавить `resolution` в перечень запрашиваемых полей n8n: `fields: "summary,issuetype,created,status,resolution"`. И передавать его в `Transform Data`.

### 1.3. Отсутствие StatusCategory
* **Где найдено:** `src/lib/utils.ts` (`isNonDoneTerminal`) и n8n.
* **Суть несостыковки:** Фронтенд пытается использовать нативное поле `statusCategory` для отсеивания терминальных статусов, но n8n его не передает. Поэтому приложение вечно упирается в хардкод-заглушку `['Отменена', 'Архив', 'Установлено']`.
* **Последствия:** Если вы добавите в Jira новый терминальный статус, приложение его не распознает. У вас начнут копиться "мертвые" задачи в метрике WIP Aging.
* **Рекомендация:** Запрашивать и передавать `statusCategory` в настройках n8n.

## 2. Логические несостыковки внутри кода (Фронтенд)

### 2.1. Игнорирование цикла возврата (Loopback) после первого релиза
* **Где найдено:** `src/lib/metrics.ts` (`calcCycleTime`).
* **Суть несостыковки:** Чтобы учесть откаты из тестирования (петли качества), код ищет *последнее* вхождение в `ctStart` (например, Разработка). Однако конечное время жестко фиксируется по **первому** вхождению в `ctEnd` (Готова к Prod). 
* **Конфликт:** Если задачу вернут на Разработку уже *после* того, как она хотя бы раз побывала в "Готова к Prod" (например, отменили релиз), весь этот второй цикл доработки просто не учтется в Cycle Time!
* **Рекомендация:** Рассчитывать `endTime` тоже по **последнему** входу в `ctEnd`, либо по ближайшему `ctEnd`, следующему за найденным `lastStart`.

### 2.2. Завышенный Cycle Time для User Story
* **Где найдено:** `src/lib/utils.ts` (автоопределение воркфлоу).
* **Суть несостыковки:** В `task_transitions.md` видно, что цикл "Задачи" заканчивается перед релизом (Готово к Prod). Для User Story (`userStory_transitions.md`) система технически приравнивает конец активной фазы (`ctEnd`) к самому концу жизни задачи (`ltEnd` "ГОТОВО").
* **Конфликт:** Между фазами разработки и "ГОТОВО" у US есть огромные статусы ожидания: `ЧАСТИЧНЫЙ РЕЛИЗ`, `РЕЛИЗ`, `РЕВЬЮ`. Замешивание этого времени в Cycle Time команды разработки делает метрику неэффективной (смешивается работа Delivery разработчиков и маркетологов Release).
* **Рекомендация:** Для User Story лучше определять `ctEnd` как "ГОТОВО К ПРИЕМКЕ" или переписать логику обнаружения статусов.

## 3. Ограничения Jira REST API и риски потери исторических данных

Вы просили указать, если есть другие, более подходящие API-методы в рамках [Jira REST API 9.12.7](https://docs.atlassian.com/software/jira/docs/api/REST/9.12.7/). Существующий метод является опасным.

* **Проблема `maxResults=100` для `expand=changelog`:**
При использовании глобального JQL-поиска Jira Server (`POST /rest/api/2/search?expand=changelog`), внутренний возврат `histories` всегда ограничен **100 записями на одну задачу**. Для "пинг-понга" между QA и Dev в сложных багах счетчик истории легко переваливает за 100.
* **Последствие:** Ранние переходы статусов ("Бэклог", "Идея") просто обрежутся. Lead Time для этих задач рассчитать не получится, так как стартовая дата исчезнет из массива `transitions`.

### Какие методы можно использовать:

1. **Точечный API-Pull-метод:**
Делать поиск только для того, чтобы забрать список ключей задач (`/rest/api/2/search`).
Затем для каждой задачи с длинной историей дергать точечный эндпоинт:
`GET /rest/api/2/issue/{issueId}?expand=changelog`.
*(Это обходит общие лимиты поиска, либо можно использовать `GET /rest/api/2/issue/{issueId}/changelog` с поддержкой станичной пагинации).*

2. **Push-модель (Webhooks) — Архитектурный Идеал:**
Отказаться от скачивания исторических данных "post-factum" через n8n JQL. 
Настроить в `Jira Administration > System > Webhooks` вебхук на событие `jira:issue_updated`. При любой смене статуса Jira сама моментально отправит данные в n8n на сохранение в вашу небольшую базу данных (например MongoDB ил PostgreSQL). 
*(Плюсы: Нет проблемы с лимитом истории в 100 переходов, 100% точно рассчитать старые задачи).*

---
**ИТОГ:** Из-за некорректного JQL запроса и потерянных полей API, в текущем приложении пользователи принимают решения на ложных данных. Рекомендуется приостановить эксплуатацию и немедленно перенастроить n8n webhook (поля JQL, resolution, statusCategory).
