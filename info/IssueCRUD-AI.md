Инструкция для ИИ-агента по реализации модуля "Jira Issues & AI Assistant"
1. Контекст проекта и архитектурные ограничения (Project Context)
Ты работаешь над клиентским веб-приложением Kanban Metrics. Это легковесный, производительный инструмент для аналитики, который должен работать максимально плавно и не перегружать систему пользователя.
Стек технологий: React 19, TypeScript (со строгой типизацией всех API-ответов), Tailwind CSS 4, Vite.
Ограничение сборки (Критично): Приложение собирается в единый, самодостаточный HTML-файл (через vite-plugin-singlefile). В связи с этим категорически запрещено использовать внешние ассеты (растровые картинки, тяжелые шрифты из локальных файлов). Все иконки должны быть импортированы как инлайн SVG (рекомендуется использовать библиотеку lucide-react, так как она отлично поддерживает tree-shaking).
Стейт-менеджмент и Хранение: Используются стандартные React Hooks (useState, useEffect, useCallback, useContext). Данные сессии и настройки API (например, Base URL для вебхуков n8n) хранятся исключительно в localStorage. При инициализации компонентов необходимо проверять наличие этих настроек и корректно обрабатывать их отсутствие (показывать fallback-UI с просьбой перейти в настройки).
2. Дизайн-система (Donezo Theme)
Приложение использует философию дизайна "Donezo" — это современный, чистый интерфейс с элементами стекломорфизма (glassmorphism), ощущением визуальной легкости, мягкими тенями, большими скруглениями и акцентными зелеными цветами. При создании новых компонентов строго соблюдай эти визуальные правила:
Цветовая иерархия: * donezo-dark: Очень темный изумрудный. Основной цвет для Primary элементов, активных табов навигации и заголовков.
donezo-primary: Яркий зеленый. Цвет взаимодействия — используется для акцентов, состояний :hover и рамок фокуса инпутов (focus ring).
donezo-light: Бледно-зеленый. Используется для мягкой подсветки (например, подложек неактивных бейджей или hover строк таблицы).
bg-gray-50 и bg-gray-100: Базовые фоны для инпутов и разделителей, создающие контраст с белоснежными карточками.
Скругления (Border Radius): Это визитная карточка темы. Кнопки — ТОЛЬКО rounded-full. Главные карточки, модальные окна и панели — rounded-3xl или rounded-[32px]. Инпуты и мелкие плашки — rounded-xl. Избегай стандартных rounded-md.
Кнопки (Интерактивность):
Primary: bg-donezo-dark text-white rounded-full hover:bg-donezo-primary hover:-translate-y-0.5 transition-all duration-200 shadow-sm. (Обязательно добавляй эффект "поднятия" при наведении).
Secondary: bg-white text-gray-700 border border-gray-200 rounded-full hover:bg-donezo-light hover:text-donezo-dark transition-all duration-200.
Инпуты и формы: Инпуты должны выглядеть "вдавленными" в неактивном состоянии и "выделяться" при фокусе: bg-gray-50 border border-gray-100 rounded-xl focus:bg-white focus:border-donezo-primary focus:ring-2 focus:ring-donezo-light transition-all duration-200.
Тени и Глубина: Используй shadow-sm для кнопок, чтобы отделить их от фона. Для крупных панелей (Slide-over) используй глубокие тени shadow-2xl или кастомный shadow-donezo, чтобы создать эффект левитации над основным контентом.
3. Описание задачи и бизнес-ценность (Epic Goal)
Необходимо расширить функционал приложения, добавив новую вкладку "Список задач" (Issues). Главная ценность этого обновления — внедрение встроенного ИИ-помощника, который избавит команду от рутинного написания и форматирования тикетов.
Требуется реализовать боковую выезжающую панель (Slide-over) для создания и редактирования задач напрямую в Jira. ИИ-помощник должен уметь генерировать задачи с нуля по короткому промпту пользователя, а также точечно оптимизировать существующий контент (summary, description), приводя его к профессиональным стандартам Agile. Весь процесс должен происходить бесшовно, без перезагрузки страницы и потери контекста (пользователь всегда должен видеть таблицу задач на заднем фоне).
4. Спецификация API (API Endpoints) и Обработка ошибок
Все запросы отправляются на базовый URL вебхуков n8n, который предварительно настраивается пользователем. Важно: Всегда предусматривай состояния загрузки (isLoading) и обработку ошибок (try/catch с выводом toast-уведомлений), так как запросы к ИИ могут занимать от 3 до 10 секунд.
A. Получение списка задач
GET /jira/issues — возвращает массив кратких объектов задач для рендера таблицы.
GET /jira/issues?key={KEY} — возвращает глубокую, детализированную информацию по одной задаче (включая массивы comments и attachments_info). Используется при открытии режима редактирования.
B. ИИ: Генерация задачи с нуля
Endpoint: POST /ai-generate
Request Body: { "issue_type": "Ошибка" | "Задача" | "User Story" | "Техдолг", "user_prompt": "Текст от пользователя" }
Response Body: Строгий JSON-объект задачи: { summary, description, priority, issuetype, checklists: [{name, checked, mandatory, rank, isHeader}] }.
UX Требование: Пока идет запрос, форма должна быть заблокирована (disabled), а пользователю должна демонстрироваться анимация загрузки (скелетон или спиннер с текстом "Нейросеть формирует задачу...").
C. ИИ: Оптимизация поля
Endpoint: POST /ai-optimize
Request Body: { "field_type": "summary" | "description", "text": "Оригинальный, неструктурированный текст" }
Response Body: { "optimized_text": "Профессионально отформатированный текст" }
D. Jira: Создание и Обновление
POST /jira/issues — Создать новую задачу. Требует передачи всех обязательных полей, сформированных на клиенте.
PATCH /jira/issues?key={KEY} — Обновить существующую задачу. Ожидает передачи только объекта updates с измененными полями, что минимизирует нагрузку на сеть.
5. Требуемая архитектура компонентов
Для поддержания чистоты кода и переиспользуемости реализуй следующие компоненты:
1. IssuesTab.tsx (Вкладка со списком)
Основной View. Содержит таблицу с колонками: Key (кликабельная ссылка), Summary, Status, Priority.
Вверху справа располагается кнопка + Создать задачу (Primary).
В строках таблицы: кнопка-иконка Редактировать (например, иконка карандаша из lucide-react).
При пустом списке необходимо показывать красивый Empty State с иллюстрацией (SVG) и призывом создать первую задачу.
Управляет состоянием видимости боковой панели (открыта/закрыта) и передает выбранный issueKey (если это режим редактирования).
2. IssueSlideOver.tsx (Выезжающая панель)
Контейнер и анимации: Оверлей bg-slate-900/20 backdrop-blur-sm (должен плавно появляться, fade-in). Сама панель должна выезжать с правой стороны экрана (анимация slide-in-right), занимая ширину max-w-4xl (или фиксированные w-[800px]).
Дизайн панели: bg-white rounded-l-[32px] shadow-2xl overflow-y-auto.
Окно должно поддерживать закрытие по клику на затемненный оверлей, по нажатию клавиши Escape или по клику на кнопку-крестик (X) в правом верхнем углу.
Внутри панели условно рендерится компонент CreateIssueForm (если issueKey отсутствует) или EditIssueForm (если передан issueKey).
3. CreateIssueForm.tsx (Режим создания)
AI-генератор (блок сверху): * Выпадающий список (Select) для жесткого выбора типа задачи (User Story, Задача, Ошибка, Техдолг).
Просторная Textarea для ввода контекста (user_prompt).
Кнопка "Сгенерировать ✨" (Primary).
Основная форма: Поля для summary (input), description (textarea), priority (select).
Сложные поля: Блок checklists (список инпутов с возможностью добавить новый пункт через кнопку "+ Добавить" и удалить существующий через иконку корзины) и labels (реализовать ввод тегов — при нажатии Enter текст превращается в зеленый бейдж).
Кнопка "Создать в Jira" фиксируется в самом низу формы (Sticky footer) и отправляет итоговый payload.
4. AiSummaryInput.tsx (Инлайн оптимизация)
Умная замена обычного инпута для поля summary.
Справа внутри поля расположена иконка ✨.
Взаимодействие: При клике на ✨ инпут блокируется, иконка заменяется на маленький спиннер. Отправляется запрос POST /ai-optimize.
Результат: После получения ответа старый текст плавно заменяется на новый. Инпут на 2 секунды подсвечивается акцентной рамкой (ring-2 ring-donezo-primary).
Важно: Снизу появляется небольшая текстовая кнопка "Отменить (Undo)", позволяющая пользователю моментально вернуть оригинальный текст, если результат нейросети его не устроил.
5. AiDescriptionDiff.tsx (Side-by-Side оптимизация)
Продвинутый компонент для поля description, минимизирующий риск потери данных.
Начальное состояние: Обычная редактируемая textarea + кнопка "Оптимизировать ✨".
Состояние Diff (после ответа ИИ): Интерфейс плавно перестраивается в двухколоночную сетку (grid grid-cols-2 gap-4).
Левая колонка (Оригинал): Оригинальный текст пользователя. Заблокирована для редактирования (disabled, bg-gray-50, text-gray-500). Служит как безопасный референс.
Правая колонка (ИИ-результат): Оптимизированный текст (bg-white, активная textarea). Пользователь может внести ручные правки прямо здесь перед сохранением.
Под правой колонкой располагаются две кнопки: "Принять изменения" (Primary) и "Отклонить" (Secondary). При принятии интерфейс сворачивается обратно в одну колонку с новым текстом.
6. EditIssueForm.tsx (Режим редактирования)
При маунте компонента делает запрос GET /jira/issues?key=... для получения полных данных задачи и показывает глобальный лоадер.
Интегрирует в себя умные компоненты AiSummaryInput и AiDescriptionDiff.
Реализует логику "Dirty State": отслеживает, какие именно поля были изменены пользователем по сравнению с начальными данными. Кнопка "Сохранить изменения" активна только если есть изменения.
При сохранении отправляет PATCH /jira/issues?key=..., передавая строго объект updates только с модифицированными ключами.
6. План действий и последовательность разработки (Step-by-Step)
Для обеспечения стабильности, следуй этому алгоритму разработки:
Модели и Типы: Начни с создания файла types/jira.ts. Опиши строгие TypeScript интерфейсы (IssueShort, IssueDetailed, ChecklistItem, запросы к ИИ) на основе спецификации OpenAPI.
Скелет UI: Реализуй визуальный компонент IssueSlideOver с правильными Tailwind-классами анимации и стекломорфизма. Убедись, что закрытие работает корректно.
Список задач: Создай компонент IssuesTab с фейковым (моковым) массивом данных. Проверь логику открытия Slide-over панели при клике на кнопки создания и редактирования.
Хуки API: Вынеси логику запросов в кастомные хуки (например, useJiraIssues, useAIAssistant), используя fetch. Реализуй в них обработку ошибок и индикацию загрузки.
Форма создания: Собери CreateIssueForm. Сначала реализуй ручной ввод и добавление чек-листов/тегов. Затем привяжи логику ИИ-генератора к верхней части формы.
Умные компоненты: Разработай изолированные компоненты AiSummaryInput (с логикой Undo) и AiDescriptionDiff (с логикой двух колонок). Протестируй их независимо.
Интеграция редактирования: Собери EditIssueForm, внедри в нее умные компоненты, реализуй проверку измененных полей (dirty state) и функцию PATCH-обновления.
Финальный аудит: Убедись, что код разбит на небольшие функции, нет дублирования логики, и везде используются иконки из lucide-react.
