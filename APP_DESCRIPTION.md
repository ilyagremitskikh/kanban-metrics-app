# Kanban Metrics — Описание приложения

## Назначение

**Kanban Metrics** — клиентское веб-приложение для анализа эффективности команды по данным из Jira. Инструмент рассчитывает ключевые метрики потока, строит прогнозы и помогает командам принимать data-driven решения в процессе Kanban-поставки.

Приложение работает как единый HTML-файл (~450 КБ), не требует бэкенда и разворачивается на любом статическом хостинге или открывается напрямую из файловой системы.

---

## Ключевые концепции

Приложение работает с двумя типами данных:

- **Исторические данные** — закрытые задачи со временем прохождения через статусы (Lead Time, Cycle Time)
- **Текущий снимок** — активные задачи в работе (WIP, Aging WIP)

Весь workflow разбит на два потока:

| Поток | Статусы (примеры) | Смысл |
|-------|-------------------|-------|
| **Upstream** | Анализ, Подготовка, Оценка риска | Подготовка задачи к разработке |
| **Downstream** | Разработка, Code Review, Тестирование | Непосредственная реализация |

## Бизнес-функции

### 1. Метрики потока (вкладка Metrics)

#### Lead Time (LT)
Полное время жизни задачи — от создания до перехода в статус «Готово». Показывает, насколько быстро команда доставляет ценность с момента возникновения идеи.

#### Dev Cycle Time (CT)
Время активной разработки — от первого перехода в downstream-статус до «Готово». Отражает производительность команды без учёта времени ожидания в очереди и анализа.

#### Upstream Time
Время на подготовку задачи — от входа в upstream-статус до начала разработки. Помогает выявлять узкие места на этапе анализа и планирования.

Для каждой метрики рассчитываются **перцентили P50, P85, P95** по всем закрытым задачам. Данные отображаются в виде scatter chart с линиями перцентилей.

---

### 2. Throughput (пропускная способность)

Количество задач, завершённых командой за каждую рабочую неделю. Визуализируется в виде столбчатой диаграммы.

Источник данных:
- **По умолчанию**: рассчитывается из истории переходов статусов (changelog)
- **Опционально**: нативные даты закрытия из Jira (через отдельный webhook)

Throughput используется как основа для прогнозирования.

---

### 3. WIP (Work In Progress)

Текущее количество задач в работе — прямой подсчёт задач в активных статусах на момент загрузки данных. Отдельно отображаются:

- **Upstream WIP** — задачи на стадии анализа и подготовки
- **Downstream WIP** — задачи на стадии разработки и тестирования

---

### 4. Aging WIP

Задачи в работе с индикатором «возраста» — сколько времени каждая задача уже находится в текущем статусе. Цветовая индикация:

| Цвет | Порог | Значение |
|------|-------|---------|
| Зелёный | < P50 | Норма |
| Жёлтый | P50 — P85 | Предупреждение |
| Красный | > P85 | Критично, задача устаревает |

Пороговые значения рассчитываются из исторического Cycle Time. Aging WIP показывается отдельно для upstream и downstream задач.

---

### 5. Predictability Score (предсказуемость)

Интегральная оценка стабильности команды (0–100 баллов).

Формула:
```
CV = StdDev(CT) / Mean(CT)
Score = max(0, 100 − CV × 25)
```

Чем ниже вариабельность Cycle Time, тем выше оценка. Дополнительно рассчитывается:

- **SLE Hit Rate** — процент задач, завершённых в пределах P85 (Service Level Expectation)
- **Исторический тренд** — sparkline за последние 12 недель

Ориентир: Score ≥ 70 соответствует зрелой предсказуемой команде.

---

### 6. Monte Carlo (прогнозирование)

10 000 симуляций на основе исторического throughput. Поддерживает три режима:

#### Режим «Сколько задач → когда?»
Ввод: количество задач
Результат: P50/P85/P95 — прогноз по неделям до завершения

#### Режим «До даты → сколько задач?»
Ввод: целевая дата
Результат: P50/P85/P95 — прогноз количества задач, которые успеют завершиться

#### Режим «Очередь» (приоритизированный бэклог)
Ввод: список задач в порядке приоритета
Результат: индивидуальный прогноз даты завершения (P50/P85/P95) для каждой задачи с учётом задач в работе (WIP)

Режим «Очередь» интегрирован с блоком RICE: задачи отправляются в прогноз прямо из таблицы приоритизации.

Результаты отображаются в виде гистограммы с цветовой разметкой перцентилей.

---

### 7. RICE Scoring (приоритизация задач)

Оценка и ранжирование бэклога по трём типам задач:

#### User Story / Task — фреймворк RICE
```
Score = (Reach × Impact × (Confidence / 100)) / Effort
```

| Параметр | Описание | Допустимые значения |
|----------|----------|---------------------|
| Reach | Охват пользователей | Число |
| Impact | Величина эффекта | 0.25 (мин.) → 3 (масс.) |
| Confidence | Уверенность в оценке | 25%, 50%, 80%, 100% |
| Effort | Трудоёмкость | 0.5–40 недель |

#### Bug — матрица Severity × Priority
```
Score = Severity_weight × Priority_weight  (макс. 16)
```

Уровни: Critical / High / Medium / Low
Цветовая индикация: красный ≥ 12, оранжевый ≥ 8, жёлтый ≥ 4, серый < 4

#### Tech Debt
Оценивается по **Cost of Delay** — числовой показатель стоимости откладывания задачи.

Оценки синхронизируются обратно в Jira через n8n. Задачи из RICE-таблицы можно одним кликом отправить в прогнозирование Monte Carlo (режим «Очередь»).

---

### 8. Таблица задач

Полный список загруженных задач с метриками. Поля:

- Ключ задачи (кликабельная ссылка на Jira)
- Название
- Тип задачи
- Lead Time (дни)
- Dev Cycle Time (дни)
- Upstream Time (дни)
- Статус
- Дата завершения

Поддерживается сортировка по любому столбцу.

---

### 9. Настройки (вкладка Settings)

- **URL основного webhook** — n8n для получения метрик (обязательно)
- **URL webhook пропускной способности** — отдельный источник throughput (опционально)
- **Режим запроса**: стандартный (project key + тип задач) или custom JQL
- **Фильтр типов задач** — выбор типов для включения в анализ

Все настройки сохраняются в `localStorage` браузера.

## Архитектура данных

```
Браузер (React SPA)
    │
    ├── POST /webhook/kanban-metrics
    │       ↓ project key / JQL
    │   n8n → Jira API (задачи + история переходов)
    │       ↓
    │   {issues: [{key, created, currentStatus, transitions: [{to, date}]}]}
    │
    ├── GET /webhook/rice-scoring (опционально)
    │       ↓
    │   n8n → Jira + DataTable
    │       ↓
    │   {issues: [{key, reach, impact, confidence, effort, severity, ...}]}
    │
    └── POST /webhook/rice-score-update
            ↓ обновлённые оценки
        n8n → Jira DataTable
```

Расчёт всех метрик выполняется **в браузере** на основе полученных данных. Никакие вычисления на стороне сервера не требуются.

## Развёртывание

Приложение собирается в единый `index.html` (~450 КБ). Работает:

- Напрямую из файловой системы (`file://`)
- С любого статического HTTP-сервера (nginx, Apache и др.)
- Из Docker-контейнера

Текущее development-окружение: n8n на `n8n.mindhackerdev.ru`, приложение разворачивается на домашний сервер через `./deploy.sh`.
