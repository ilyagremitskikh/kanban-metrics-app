# Дизайн: подзадачи и иерархия задач в форме создания/редактирования

## Контекст

Сейчас форма создания и редактирования задач покрывает базовые поля Jira-задачи, ручной чеклист и AI-помощь для summary/description. При этом в рабочем процессе не хватает двух вещей:

1. Нельзя создавать дочерние элементы прямо из формы текущей задачи.
2. В общем списке задач не видно связи с родителем и эпиком.

Отдельно подтверждено, что AI-генерация чеклистов оказалась избыточной и должна быть удалена. Ручное редактирование чеклиста при этом остается.

API уже отдает эпик-связь в данных задачи через `epic_key` и `epic.key`. Для дочерней связи используется `parent`, а для формы редактирования целевой detail endpoint должен уметь вернуть дочерние элементы текущей задачи.

## Цели

- Добавить создание дочерних элементов прямо из формы редактирования задачи.
- Для обычной задачи разрешить создание только Jira `Подзадачи`.
- Для `Epic` разрешить создание обычного тикета выбранного типа, привязанного к эпику, но не являющегося подзадачей.
- После создания дочернего элемента оставлять пользователя в форме родителя и показывать обновленный список детей.
- Удалить AI-автогенерацию чеклистов из формы создания и редактирования.
- Показать в таблице задач родителя для подзадач и принадлежность к эпику для всех типов.

## Не цели

- Не добавляем редактирование `parent` или `epic` у уже существующей задачи.
- Не добавляем массовые операции с дочерними задачами.
- Не уводим создание дочерних задач в отдельный экран или полноценный мастер.
- Не меняем существующий AI-поток для генерации summary/description, кроме удаления чеклистов из результата.

## Подходы

### Подход 1. Встроенный дочерний блок в форму

Внизу `CreateIssueForm` и `EditIssueForm` появляется блок дочерних задач со списком и вложенной формой создания.

Плюсы:
- пользователь не теряет контекст родительской задачи;
- логика `Epic` и обычной задачи живет в одном месте;
- после создания можно сразу показать обновленный список детей.

Минусы:
- форма становится сложнее;
- нужно аккуратно разделить состояние основной формы и вложенного создания.

### Подход 2. Отдельная модалка для создания дочерней задачи

Форма родителя открывает вторичную модалку или sheet для создания дочернего элемента.

Плюсы:
- меньше нагрузки на основную форму;
- проще изолировать код.

Минусы:
- более рваный UX;
- слабее ощущается связь с родителем;
- придется дополнительно синхронизировать список созданных детей.

### Подход 3. Упрощенные inline-строки без формы

Показываются только быстрые поля для summary и type, без нормального редактора.

Плюсы:
- минимальные изменения интерфейса.

Минусы:
- не покрывает реальные поля дочерней задачи;
- быстро упрется в ограничения по описанию, меткам и чеклисту;
- хуже согласуется с текущим UI-паттерном.

### Выбранный подход

Берем подход 1. Он лучше всего соответствует текущей структуре приложения, где создание и редактирование уже организованы вокруг полноценных форм, и дает самый естественный UX.

## UX и поведение формы

### Общая модель

В нижней части формы появляется новый блок `Дочерние задачи`.

Поведение блока зависит от типа текущей задачи:

- если текущая задача не `Epic`, показывается кнопка `Добавить подзадачу`;
- если текущая задача `Epic`, показываются выбор типа создаваемой задачи и кнопка `Добавить задачу в эпик`.

После нажатия раскрывается вложенная форма создания дочернего элемента. Пользователь не покидает текущую форму.

### Ограничение для CreateIssueForm

Пока родительская задача еще не создана в Jira, дочерние элементы создать нельзя.

Для `CreateIssueForm` блок `Дочерние задачи`:

- виден всегда;
- находится в disabled-состоянии до первого сохранения родителя;
- показывает понятный текст: `Сначала сохраните задачу, потом можно будет добавить дочерние элементы`.

Это одинаково относится и к созданию обычной задачи, и к созданию нового эпика.

### Поведение для обычной задачи

Из обычной задачи можно создать только Jira `Подзадачу`.

Пользователь не выбирает тип вручную: он фиксирован и равен `Подзадача`.

Во вложенной форме доступны поля:

- `summary`;
- `description`;
- `priority`;
- `labels`;
- `checklist`.

Связи руками не редактируются. При сохранении фронт автоматически отправляет:

- `issuetype = 'Подзадача'`;
- `parentKey = currentIssue.key`;
- `epicKey = currentIssue.epic_key`, если у родителя есть эпик.

### Поведение для Epic

Из `Epic` создается не подзадача, а обычный тикет выбранного типа, принадлежащий эпику.

Во вложенной форме доступны поля:

- `issuetype` из набора допустимых не-epic типов;
- `summary`;
- `description`;
- `priority`;
- `labels`;
- `checklist`.

При сохранении фронт автоматически отправляет:

- `epicKey = currentEpic.key`;
- без `parentKey`.

### После успешного создания дочернего элемента

После успешного создания:

- родительская форма остается открытой;
- вложенная форма сбрасывается;
- список дочерних элементов обновляется;
- пользователь сразу видит новый элемент в блоке `Дочерние задачи`.

### Список дочерних элементов в форме

В блоке `Дочерние задачи` под кнопкой создания показывается компактный список уже созданных детей.

В каждой строке отображаются:

- `key`;
- `type`;
- `summary`;
- `status`;
- краткая индикация связи.

Индикация связи:

- для дочерних элементов обычной задачи показывается бейдж `Подзадача`;
- для тикетов внутри эпика показывается бейдж `В эпике`.

На первом этапе редактирование дочерних элементов из этого списка не добавляется. Допустима только ссылка/кнопка открытия задачи в Jira.

## Модель данных и API-контракт

### Расширение фронтовых типов

`JiraIssueShort` и `JiraIssueDetailed` должны поддерживать поля иерархии как optional:

- `parent?: { key: string } | null`;
- `parent_key?: string | null`;
- `epic?: { key: string } | null`;
- `epic_key?: string | null`.

Для detail-ответа дополнительно нужен список дочерних элементов:

- `children?: JiraIssueShort[]`.

Все новые поля остаются optional, чтобы не ломать старые снапшоты и неполные ответы.

### Контракт создания

`CreateIssueRequest` расширяется опциональными полями:

- `parentKey?: string`;
- `epicKey?: string`.

Фронт использует их так:

- создание из обычной задачи: `issuetype = 'Подзадача'`, `parentKey = currentIssue.key`, `epicKey = currentIssue.epic_key ?? undefined`;
- создание из эпика: `issuetype = selectedType`, `epicKey = currentEpic.key`, `parentKey` не передается.

### Контракт редактирования

`UpdateIssueRequest` в рамках этой задачи не расширяется полями для изменения иерархии.

Причина:

- это отдельная область бизнес-логики;
- она повышает риск случайного разрушения структуры задач;
- она не нужна для согласованного UX текущего сценария.

### Источник children

Предпочтительный вариант: detail endpoint формы редактирования сразу возвращает `children` для текущей задачи.

Fallback-поведение допустимо только временно:

- если `children` отсутствует, форма показывает пустой список без ошибки;
- построение детей фильтрацией общего списка задач допускается только как временный обходной путь, если расширение n8n-флоу откладывается.

## Изменения AI и чеклистов

### Что убираем

Из UI и фронтовой логики удаляется AI-автогенерация чеклистов:

- кнопка генерации чеклиста в `ChecklistEditor`;
- состояние загрузки и ошибки AI-чеклиста;
- вызов `aiChecklist`;
- зависимость от `WEBHOOK_PATHS.aiChecklist`, если она больше нигде не используется;
- любые тексты, связанные с генерацией чеклистов.

### Что остается

Чеклист остается полностью ручным:

- добавление пунктов;
- редактирование;
- удаление;
- передача в create/update Jira issue.

AI-черновик задачи в `CreateIssueForm` может остаться, но без `checklists` в response-модели и без автоматической подстановки чеклиста.

## Таблица задач

### Новые колонки

В `IssuesTab`/`TasksDataTable` добавляются две колонки:

- `Родитель`;
- `Эпик`.

### Правила отображения

- для `Подзадачи` колонка `Родитель` показывает `parent_key`, если он есть;
- для обычной задачи внутри эпика колонка `Родитель` показывает `—`;
- для любого типа колонка `Эпик` показывает `epic_key`, если он есть;
- для самого эпика и задач вне эпика колонка `Эпик` показывает `—`.

На первом этапе сортировка по этим колонкам не требуется. Приоритет у читаемого отображения без перегрузки таблицы.

## Ошибки и деградация

- Если у вложенного create-запроса произошла ошибка, пользователь остается в родительской форме и видит локальную ошибку внутри блока `Дочерние задачи`.
- Если detail endpoint не вернул `children`, форма не должна падать.
- Если тип `Подзадача` не пришел в `availableTypes`, фронт все равно должен уметь создать подзадачу как фиксированный системный тип.
- В epic-сценарии список допустимых типов не должен включать `Epic`.

## Тестирование

### Проверки поведения

- редактирование обычной задачи позволяет создать `Подзадачу`;
- новая подзадача появляется в списке детей без ухода со страницы;
- если у родителя есть `epic_key`, созданная подзадача наследует эпик;
- редактирование `Epic` позволяет создать обычную задачу выбранного типа внутри эпика;
- созданная задача внутри эпика появляется в списке дочерних элементов;
- `CreateIssueForm` показывает disabled-блок дочерних задач до первого сохранения;
- чеклист работает только вручную;
- таблица задач корректно показывает `Родитель` и `Эпик`.

### Регрессионные проверки

- создание обычной задачи без дочерних элементов не ломается;
- редактирование summary, description, priority, labels и checklist не ломается;
- существующий AI-флоу для summary/description продолжает работать;
- бизнес- и downstream-вкладки продолжают корректно фильтровать задачи.

## Критерии готовности

Фича считается завершенной, когда выполняются все условия:

1. Из формы редактирования обычной задачи можно создать Jira `Подзадачу`.
2. Из формы редактирования `Epic` можно создать обычный тикет выбранного типа, привязанный к эпику.
3. После создания дочернего элемента пользователь остается в форме родителя и видит обновленный список детей.
4. В форме создания новой задачи блок дочерних элементов виден, но заблокирован до первого сохранения.
5. AI-автогенерация чеклистов полностью удалена из фронта.
6. В таблице задач видны колонки `Родитель` и `Эпик`.
7. Существующие сценарии создания и редактирования задач продолжают работать без регрессий.
